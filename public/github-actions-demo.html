Using TicTrack as a Demo Project

What is GitHub Actions?

GitHub Actions is an automation platform that lets you build, test, and deploy your code directly from your GitHub repository. Think of it as a robot that runs your checklist every time you push code — catching bugs, running tests, and ensuring nothing breaks.

Why GitHub Actions?

Automatic Quality Checks

Every push automatically runs tests. If something breaks, you know immediately — not after it reaches users.

Saves Developer Time

No more manually running tests or deployments. The robot handles it while you focus on building features.

Multi-Version Testing

Test your code against multiple Node.js versions (18.x, 20.x, 22.x) simultaneously — ensuring compatibility.

Deploy With Confidence

Passing CI checks mean every deploy is backed by verified tests. No more "it works on my machine."

Instant Feedback

Pull requests show check statuses directly. Reviewers see green checks before merging — catching issues early.

This Project's Workflows

This project has two GitHub Actions workflows. Switch between them below.

name: Todo
on:
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]

jobs:
  build:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [22.x]

    steps:
    - uses: actions/checkout@v4
    - name: Use Node.js ${{ matrix.node-version }}
      uses: actions/setup-node@v4
      with:
        node-version: ${{ matrix.node-version }}
        cache: 'npm'
    - name: Install dependencies
      run: npm ci
    - name: Run tests
      run: npm test --if-present
    - name: Start application and verify visibility
      run: |
        npm run dev &
        npx wait-on http://localhost:3000 --timeout 30000
        curl -f http://localhost:3000
1

Trigger

Runs on every push or pull_request to the main branch. This protects the production branch from broken code.

2

Environment & Dependencies

Runs on ubuntu-latest with Node.js 22.x. Installs deps via npm ci (clean install) with cache: 'npm' for faster runs.

3

Test & Verify

Runs npm test (if any tests exist). Then starts the dev server, waits for port 3000, and curls it to confirm the app is actually serving requests.

  • Easy to forget testing before deploying

What Gets Checked — In Context

  • Dependencies aren't broken

    npm ci installs exactly what's in package-lock.json. If a dependency is missing or incompatible, the pipeline fails.

  • Tests pass before merge

    npm test runs all tests. For TicTrack, this means the API endpoints, Excel operations, and task management logic are verified.

  • App boots correctly

    The todo.yml workflow starts the dev server and curls it. If the Express server crashes on startup, the pipeline catches it.

  • Multiple Node versions work

    node.js.yml tests on 18.x, 20.x, 22.x. If someone deploys on Node 22 but users run 18, the matrix catches the mismatch.

  • Deployment confidence

    Combined with Vercel & Render configs (see vercel.json and render.yaml), passing CI means you can deploy with one click.

Can We Use Docker in GitHub Actions? Absolutely

Docker and GitHub Actions are not alternatives — they solve different problems. GitHub Actions is the automation engine, Docker is the packaging format. You can (and often do) use Docker inside GitHub Actions:

Without Docker in CI

  • npm ci installs deps directly on Ubuntu runner
  • Runs npm test with the system Node.js
  • Works fine for simple Node.js apps

This is what TicTrack does today.

With Docker in CI

  • docker build packages your app into an image
  • Tests run inside the container (same env as prod)
  • Same image deploys to staging & production

Needed when you have system deps or deploy to container runtimes.

Verdict for TicTrack: Docker is optional. The current setup (direct npm ci + npm test on Ubuntu) is perfectly adequate. You'd add Docker when you need to ship the app to Docker-based platforms or need exact environment parity.

GitHub Actions vs Jenkins — Why We Chose GitHub

Jenkins is a powerful, mature CI/CD server. But for a project like TicTrack, GitHub Actions is the better fit. Here's why:

GitHub Actions

  • Zero infrastructure — runs on GitHub's servers, no setup
  • Free for public repos — 2000 min/month for private
  • YAML in repo — config is version-controlled alongside code
  • Native GitHub integration — PR checks, status badges, auto-trigger
  • Marketplace ecosystem — 1000s of pre-built actions
  • No maintenance — no server to patch or plugins to update

Jenkins

  • Requires a server (or Docker container) to host
  • Plugins need manual updates & compatibility checks
  • Pipeline config (Jenkinsfile/Groovy) is separate from repo
  • More flexible — custom plugins, complex pipelines, any OS/agent
  • Better for large enterprises — fine-grained access control, audit trails
  • Runs on your hardware — good for compliance/air-gapped environments

Bottom line: For a solo dev project hosted on GitHub, GitHub Actions is the obvious choice — zero setup, built-in integration, and does everything this project needs. Jenkins shines in enterprise environments with custom infrastructure, compliance requirements, or complex multi-platform builds.

PHASE 2

Containerization & Orchestration

After CI validates your code, the next step is packaging it into a Docker container and deploying it on Kubernetes. Here's how this would work for TicTrack.

What is Docker?

Docker packages your application and all its dependencies into a lightweight, portable container. A container includes your code, Node.js, all npm packages, and configuration — so it runs identically on your laptop, a test server, or in production.

For TicTrack, a Dockerfile would look like this:

# Use official Node.js 20 image
FROM node:20-alpine

WORKDIR /app

# Copy package files & install
COPY package*.json ./
RUN npm ci --only=production

# Copy app source
COPY . .

# Expose port & start
EXPOSE 3000
CMD ["node", "server.js"]

Single Image

Everything your app needs in one package — Node 20, dependencies, code.

Lightweight

Alpine Linux base keeps the image under 150MB, much smaller than a full VM.

Reproducible

Same image runs everywhere — no more "it works on my machine" issues.

What is Kubernetes (K8s)?

Kubernetes is an orchestration platform that manages your containers at scale. It handles deployment, scaling, load balancing, self-healing, and rolling updates — so you don't have to.

For TicTrack, the K8s manifests would be:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: tictrack-deployment
  labels:
    app: tictrack
spec:
  replicas: 3
  selector:
    matchLabels:
      app: tictrack
  template:
    metadata:
      labels:
        app: tictrack
    spec:
      containers:
      - name: tictrack
        image: ghcr.io/yourorg/tictrack:latest
        ports:
        - containerPort: 3000
        env:
        - name: SUPABASE_URL
          valueFrom:
            secretKeyRef:
              name: tictrack-secrets
              key: supabase-url
1

Declarative Desired State

You declare "I want 3 replicas running the tictrack:latest image." K8s ensures the cluster matches that state at all times.

2

Self-Healing

If a pod crashes, Kubernetes automatically replaces it. If a node goes down, pods are rescheduled on healthy nodes.

3

Secrets Management

Sensitive values like SUPABASE_URL are pulled from K8s Secrets — never baked into the image.

apiVersion: v1
kind: Service
metadata:
  name: tictrack-service
spec:
  selector:
    app: tictrack
  ports:
  - protocol: TCP
    port: 80
    targetPort: 3000
  type: LoadBalancer
1

Stable Network Entry Point

The Service exposes your 3 pods under a single DNS name on port 80. Traffic is load-balanced across all replicas.

2

Load Balancer

type: LoadBalancer provisions a cloud load balancer (e.g. AWS ALB) that distributes incoming traffic to healthy pods.

3

Service Discovery

Other services in the cluster can reach TicTrack via http://tictrack-service — no hardcoded IPs needed.

Why Docker & K8s for TicTrack?

Without Containers Fragile

  • "Works on my machine" bugs between dev and production
  • Manual server setup for each environment
  • No auto-scaling — crashes under traffic spikes
  • Downtime during manual deploys
  • No self-healing if the app crashes

With Docker + K8s Robust

  • Identical environment everywhere — same image
  • One-command deploy via kubectl apply
  • Auto-scales from 3 to 30 replicas under load
  • Rolling updates with zero downtime
  • Pods auto-restart on failure — always available

End-to-End Architecture

Here's how everything connects — from your keyboard to a running Kubernetes cluster:

You push code to GitHub (main branch)
GitHub Actions triggers todo.yml & node.js.yml
CI runs: Checkout → Setup Node → npm ci → npm test → Verify
Docker builds the image (node:20-alpine base + app code)
Image pushed to GitHub Container Registry (ghcr.io)
Kubernetes pulls the new image & performs a rolling update
TicTrack is live with 3 replicas, load-balanced, and self-healing

Bonus: A Workflow That Builds & Deploys

Here's what the deploy.yml workflow would look like to automate the full Docker + K8s pipeline:

name: Docker Build & Deploy to K8s
on:
  push:
    branches: [ "main" ]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - name: Build Docker image
      run: docker build -t ghcr.io/yourorg/tictrack:${{ github.sha }} .
    - name: Push to registry
      run: docker push ghcr.io/yourorg/tictrack:${{ github.sha }}
    - name: Deploy to Kubernetes
      run: kubectl set image deployment/tictrack-deployment tictrack=ghcr.io/yourorg/tictrack:${{ github.sha }}
1

Build with Git SHA tag

Each build is tagged with the commit SHA (${{ github.sha }}) for traceability — you always know which commit is running.

2

Push to Container Registry

The image is pushed to GitHub Container Registry (or Docker Hub, ECR, etc.). Always authenticate via GITHUB_TOKEN.

3

Rolling Update via kubectl

kubectl set image triggers a rolling update — K8s slowly replaces old pods with new ones, maintaining zero downtime.
